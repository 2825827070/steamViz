## 1 背景及功能概要

### 1.1 背景

Steam 是全球最大的 PC 游戏数字发行平台，截至 2024 年已收录超过 10 万款游戏，并积累了千万级别的用户评论数据。这些数据蕴含着丰富的商业洞察：哪些游戏类型最受欢迎？价格与好评率之间是否存在规律？近年来游戏发行数量的变化趋势如何？然而 Steam 官方平台缺乏面向分析的聚合视图，用户若想从宏观角度理解这些数据，需要借助额外的工具。

本项目以此为出发点，设计并实现了一个桌面端 Steam 游戏数据可视化分析系统（SteamViz）。系统能够**自动从 Steam 官方 API 和 SteamSpy 第三方数据服务采集数据**，将原始的 JSON 接口响应清洗、持久化至本地 SQLite 数据库，并通过四类交互式图表（折线图、饼图、柱状图等）以及两个数据表格对外呈现，帮助用户快速获取 Steam 生态的全景视图。

项目在技术实现上追求**工程化**而非玩具化：采用多线程并发的爬取策略以提升吞吐量、使用 WAL 模式的 SQLite 解决读写并发问题、利用 Win32 原生 API 实现无边框窗口及 Aero Snap，同时实现了完整的亮/暗主题切换。

### 1.2 功能概要

系统由四个功能页面组成，通过左侧导航树切换：

| 页面 | 功能描述 |
|---|---|
| **数据概览** | 四张 Matplotlib 图表（年度发行趋势折线图、类型分布饼图、价格区间柱状图、评分等级饼图）及四个实时统计卡片 |
| **游戏列表** | 展示数据库中所有游戏，支持名称搜索、评分筛选、列标题点击排序、分页浏览，右键菜单可跳转至评价页或删除游戏 |
| **用户评价** | 展示采集到的用户评论文本，支持按游戏名过滤，推荐/不推荐以颜色区分，分页浏览 |
| **数据管理** | 提供批量采集（指定数量）和关键词搜索采集两种模式，实时进度条和日志，支持随时中止 |

## 2 可视化系统设计

### 2.1 概要设计

#### 2.1.1 整体架构

系统采用经典的**三层架构**，各层职责清晰：

```
┌─────────────────────────────────────────────────────┐
│                  表现层（UI Layer）                   │
│  MainWindow  VisualHubPage  DataListPage             │
│  ReviewListPage  DataPage   MatplotlibWidget         │
└──────────────────────┬──────────────────────────────┘
                       │ 数据请求 / 信号回调
┌──────────────────────▼──────────────────────────────┐
│                 数据访问层（Data Layer）               │
│  SteamDatabase（SQLite 封装）                         │
│  – 线程安全读写，WAL 模式                             │
└──────────────────────┬──────────────────────────────┘
                       │ 批量写入
┌──────────────────────▼──────────────────────────────┐
│                 采集层（Crawler Layer）                │
│  SteamCrawler（同步 HTTP 客户端）                     │
│  CrawlerThread（QThread 后台线程）                    │
│  – ThreadPoolExecutor 并发补充详情                   │
└─────────────────────────────────────────────────────┘
```

- **表现层** 负责所有 Qt 控件的构建、样式和主题切换，通过调用数据层接口获取数据后驱动 Matplotlib 图表或 QTableWidget 渲染。
- **数据访问层** 封装了对 SQLite 数据库的所有读写操作，使用线程锁（`threading.Lock`）和线程本地存储（`threading.local`）保证多线程环境下的安全性。
- **采集层** 运行于独立的 QThread 中，通过 `pyqtSignal` 与主线程通信，内部使用 `ThreadPoolExecutor` 实现对单个游戏详情的并发补充。

#### 2.1.2 数据流图

```
Steam Store API  ─┐
SteamSpy API     ─┤──► SteamCrawler ──► CrawlerThread ──► SteamDatabase ──► 图表/表格
Steam Review API ─┘      (HTTP)           (QThread)          (SQLite)
```

采集流程分为两个阶段：
1. **阶段一（快速批量）**：从 SteamSpy `/api.php?request=all&page=N` 接口每页批量获取 1000 款游戏的核心指标（价格、评分、游玩时长等），过滤掉已采集过的 appid，将剩余新游戏以每批 100 条写入数据库，快速填充游戏池。
2. **阶段二（并发补充）**：对入库的游戏使用 `ThreadPoolExecutor`（`max_workers=2`）并发请求 Steam Store API 获取类型、描述等详情，同时请求 Steam Review API 和 SteamSpy 补充评论及标签，最终回写数据库完成完整同步。

- **数据驱动**：所有图表数据均从数据库实时查询，切换页面或刷新时重新渲染，保证展示最新状态。

#### 2.1.5 窗口系统设计（无边框 + Windows 原生体验）

为提升视觉品质，系统实现了自定义无边框窗口，同时保留了完整的 Windows 窗口管理体验。核心思路：

1. 保留 `WS_THICKFRAME + WS_CAPTION` 等 Windows 原生窗口标志，让操作系统识别此窗口为可拖动/可 Snap 的标准窗口。
2. 使用 `DwmExtendFrameIntoClientArea(hwnd, MARGINS(-1,-1,-1,-1))` 将 DWM 合成层扩展到整个客户区，隐藏原生标题栏视觉。
3. 拦截 `WM_NCCALCSIZE` 消息，返回 0 使客户区覆盖整个窗口。
4. 拦截 `WM_NCHITTEST` 消息，自定义边缘检测逻辑（8px 热区），并分别返回 `HTCAPTION`、`HTLEFT`、`HTRIGHT` 等标准命中测试值，使 Aero Snap、双击最大化、调整大小等原生功能完全正常工作。

#### 3.1.2 统计卡片计数动画（`visual_hub.py`）

数据概览页的四个统计卡片在刷新时会播放一个缓出（ease-out cubic）的数字滚动动画，利用 QTimer 单次触发和闭包递归实现：

```python
def _animate_counter(self, key, target, fmt_fn, steps=30, interval_ms=20):
    def tick(counter=[0]):   # 利用 list 作可变默认参数传递状态
        counter[0] += 1
        t = counter[0] / steps
        t_ease = 1 - (1 - t) ** 3          # cubic ease-out 缓出曲线
        val = target * t_ease
        label = self.card_values.get(key)
        if label:
            label.setText(fmt_fn(val))
        if counter[0] < steps:
            QTimer.singleShot(interval_ms, lambda: tick(counter))
        else:
            label.setText(fmt_fn(target))   # 最后一帧精确设置终值
    QTimer.singleShot(interval_ms, lambda: tick())
```

三次缓出曲线公式 `1 - (1-t)³` 使数字在接近目标值时减速，产生弹性滚动的视觉效果，相比线性变化更具动态感。

#### 3.1.3 多数据源多阶段爬虫（`crawler.py`）

**带指数退避的网络请求封装**是爬虫稳定性的核心机制：

```python
def _request_with_retry(self, url, params=None, timeout=15, max_retries=3):
    for attempt in range(max_retries):
        if not self.is_running_func():
            return None
        try:
            resp = self.session.get(url, params=params, timeout=timeout)
            if resp.status_code == 429:  # Rate Limit
                is_store_api = "steampowered.com" in url
                base_wait = 30 if is_store_api else 5
                wait_time = (2 ** attempt) * base_wait  # 指数退避
                if not self._sleep(wait_time):           # 可中断休眠
                    return None
                continue
            resp.raise_for_status()
            return resp
        except requests.exceptions.RequestException as e:
            if attempt < max_retries - 1:
                if not self._sleep(2 ** attempt):
                    return None
```

针对 Steam 官方 API（cooldown 基数 30 秒）和 SteamSpy（cooldown 基数 5 秒）设置了差异化的等待基准，使用指数退避（`2^attempt × base_wait`）避免在高频限流情况下持续无效轮询。

**可中断休眠**保证用户点击"停止"后任务能够快速响应：

```python
def _sleep(self, duration):
    steps = int(duration / 0.1)
    for _ in range(steps):
        if not self.is_running_func():  # 每 100ms 检查一次停止标志
            return False
        time.sleep(0.1)
    return self.is_running_func()
```

**多线程并发补充详情**（`CrawlerThread.run()`）：


#### 3.1.4 线程安全的数据库封装（`database.py`）

`SteamDatabase` 需要同时被主线程（UI 查询）和爬虫线程（数据写入）访问，因此采用了两种机制保证线程安全：

```python
class SteamDatabase:
    def __init__(self, ...):
        self._lock = threading.Lock()      # 互斥锁，保护写操作
        self._local = threading.local()    # 线程本地存储，每线程独立连接
        self._init_db()

    def _get_conn(self):
        # 每个线程拥有独立的 SQLite 连接（SQLite 不能跨线程共享连接）
        if not hasattr(self._local, 'conn') or self._local.conn is None:
            conn = sqlite3.connect(self.db_path)
            conn.row_factory = sqlite3.Row       # 行结果支持字典访问
            conn.execute("PRAGMA journal_mode=WAL")    # WAL 模式允许并发读
            conn.execute("PRAGMA synchronous=NORMAL")  # 平衡安全性和性能
            conn.execute("PRAGMA busy_timeout=5000")   # 锁超时 5 秒
            self._local.conn = conn
        return self._local.conn
```

`WAL（Write-Ahead Logging）` 模式是关键：在默认的 DELETE 日志模式下，写操作会阻塞读操作；而 WAL 模式允许读写并发进行，主线程查询数据渲染图表与爬虫线程写入数据可以同时执行，大幅减少 UI 卡顿。

**批量 Upsert 操作**使用 SQLite 的 `INSERT OR REPLACE` 扩展语法：

```python
sql = f"""
    INSERT INTO games ({columns}) VALUES ({placeholders})
    ON CONFLICT(appid) DO UPDATE SET {updates}
"""
cursor.executemany(sql, [list(g.values()) for g in processed_list])
```

`ON CONFLICT(appid) DO UPDATE SET` 是 SQLite 3.24+ 支持的 UPSERT 语法，它在主键冲突时自动执行更新而非抛出异常，既保证了幂等性（重复运行不产生重复数据），也避免了先 SELECT 再 INSERT/UPDATE 的两次 IO 开销。

#### 3.1.5 无边框窗口与 Win32 消息拦截（`main_window.py`）

```python
def nativeEvent(self, eventType, message):
    retval, result = super().nativeEvent(eventType, message)
    if sys.platform != 'win32' or eventType != b'windows_generic_MSG':
        return retval, result
    try:
        msg = ctypes.cast(int(message), ctypes.POINTER(MSG)).contents

        # 拦截 WM_NCCALCSIZE: 返回 0 使客户区覆盖整个窗口，隐藏原生标题栏
        if msg.message == WM_NCCALCSIZE and msg.wParam:
            return True, 0

        # 拦截 WM_NCHITTEST: 自定义命中测试，实现边缘拖拽调整大小
        if msg.message == WM_NCHITTEST:
            dpr = self.devicePixelRatioF()
            pos = self.mapFromGlobal(QPoint(
                int(msg.pt_x / dpr), int(msg.pt_y / dpr)
            ))
            px, py = pos.x(), pos.y()
            w, h = self.width(), self.height()
            edge = 8
            if py < edge:
                if px < edge:     return True, HTTOPLEFT
                if px > w - edge: return True, HTTOPRIGHT
                return True, HTTOP
            # ... 其他方向同理
            # 顶部标题区域返回 HTCAPTION，使系统处理拖动
            if (px < 210 and py < 95) or (py < 36 and px < w - 145):
                return True, HTCAPTION
    except:
        pass
    return retval, result
```

通过 `ctypes` 直接操作 Windows 消息结构体，将 `WM_NCHITTEST` 的鼠标位置（物理像素）转换为逻辑像素坐标时，使用 `devicePixelRatioF()` 获取 DPI 缩放比例，确保在高分屏（如 2K、4K）上行为正确。

```python
    return result
```

评分等级采用与 Steam 官方一致的档位划分（好评率 ≥ 95% 为"好评如潮"，≥ 80% 为"特别好评"等），保证数据展示的准确性和用户熟悉度。

## 4 总结及体会

### 4.1 项目完成情况总结

本次课程设计顺利完成了预期目标，实现了一个功能完整的 Steam 游戏数据可视化桌面应用。主要完成内容如下：

**数据采集方面**，系统对接了三个 API 端点：Steam Store API（游戏详情）、Steam Review API（用户评论）和 SteamSpy API（销量估算、标签等非官方统计数据）。实现了两种采集模式——批量采集和关键词搜索，并针对 Steam 接口的限流机制（HTTP 429）实现了指数退避的智能重试策略，保证了采集的稳定性。

**可视化方面**，整合了 Matplotlib 的四种图表（折线图、饼图、柱状图）并通过 Qt5Agg 后端无缝嵌入 PyQt5 界面。图表支持暗/亮主题联动，并实现了数字统计卡片的缓出动画效果，提升了数据呈现的动态感。

**工程质量方面**，处理了多线程并发写入数据库的线程安全问题（`threading.Lock` + `threading.local`），使用 SQLite WAL 模式解决读写并发冲突，并通过 PyInstaller 打包成可独立分发的 Windows 桌面应用。

### 4.2 技术收获

**最大的收获是理解了 Qt 的事件循环与线程模型**。在实现爬虫功能时，最初尝试在主线程中直接执行网络请求，结果导致 UI 完全冻结——这让我深刻理解了 Qt 事件循环的单线程本质。改用 QThread 后，又遇到了"子线程不能直接操作 UI 控件"的问题，通过使用 `pyqtSignal` 跨线程发送信号、让主线程的槽函数来更新 UI，才真正解决了问题。这一过程让我彻底掌握了"线程做计算，信号传数据，槽函数改 UI"的 Qt 多线程范式。

**第二个重要收获是对数据库并发模式的理解**。默认的 SQLite 日志模式在爬虫线程写数据时会阻塞主线程的查询，导致图表刷新卡顿。切换到 WAL 模式后，读写互不阻塞，这让我理解了数据库并发控制并不是数据库领域专属的高深知识，在任何一个需要多线程读写的桌面程序中都需要认真对待。

**第三个收获是对 Windows 底层 API 的初步接触**。为了实现无边框窗口同时保留 Aero Snap、边缘拖拽调整大小等原生窗口行为，研究了 Win32 消息机制（`WM_NCHITTEST`、`WM_NCCALCSIZE`）和 DWM（Desktop Window Manager）合成层 API。用纯 Python `ctypes` 调用 C 级别的 Windows API 是很有意思的经历，也让我理解了 Qt 的窗口系统和操作系统之间的关系。

### 4.3 遇到的困难与解决方法

**困难一：Steam API 限流（Rate Limiting）**。Steam 官方接口对频繁请求会返回 HTTP 429，早期版本遇到限流会直接崩溃并丢失数据。解决方案是实现带指数退避的重试机制，并将休眠拆分为 100ms 的小段循环，使程序能在用户点击"停止"时快速响应而非卡在 sleep 中。

**困难二：SQLite 并发写入的"database is locked"错误**。多线程并发写入时频繁出现此错误。经过研究，解决方案是三管齐下：（1）开启 WAL 日志模式，减少锁的持有时间；（2）设置 `busy_timeout=5000`，在锁被占用时等待最多 5 秒而非立即报错；（3）用 `threading.local()` 为每个线程分配独立连接，避免跨线程共享连接对象（SQLite 的连接对象本身不是线程安全的）。

**困难三：Matplotlib 图表在 PyQt5 主题切换后样式不一致**。切换主题后图表内部各元素（轴标签、刻度文字、网格线）的颜色更新不完整。解决方案是在 `update_theme()` 中对 axes 上的所有可着色元素进行显式遍历更新，包括 `get_xticklabels()` 和 `get_yticklabels()` 返回的每个 Text 对象，以及四条 spine 的独立设置。

**困难四：高 DPI 显示器上鼠标命中测试偏移**。在 2K 显示器并开启 125% 缩放的环境下，`WM_NCHITTEST` 消息中的鼠标坐标是物理像素，而 Qt 的坐标系是逻辑像素，直接使用会导致命中区域偏移。解决方案是在坐标转换时除以 `devicePixelRatioF()`，获得与 Qt 坐标系一致的逻辑像素值。

### 4.4 设计反思与未来改进

回顾整个项目，有几点感到可以做得更好：

一是**数据采集效率**。当前使用 `max_workers=2` 的线程池是比较保守的设置，主要是为了不触发 Steam 的限流。如果引入更精细的令牌桶限流算法，可以在合法范围内进一步提升并发度。

二是**筛选性能**。游戏列表的筛选和排序目前在 Python 层完成（全量加载再过滤），当数据量达到数万条时性能会下降。更好的方案是将筛选条件下推到 SQL WHERE 子句中，利用数据库索引加速过滤。

三是**可视化图表的交互性**。目前 Matplotlib 图表仅支持静态展示，如果能利用 Matplotlib 的 mpl_connect 添加鼠标悬停提示（Tooltip），或者提供图表的缩放/平移功能，用户体验会更好。
